  This is the final sprint of the world redesign. It transitions the game from per-character isolated worlds to a shared world where all players explore the same generated content. It also migrates legacy characters and deploys to production.

  ## What Already Exists (Sprints 1-6)

  ### Schema (`src/server/db/schema.ts`)
  - `worlds` table: id, name, description, theme, seed, createdAt
  - `regions` table: id, worldId (FK), name, description, theme, position, connections, metadata, generatedBy
  - `areas` table: id, regionId (FK), name, description, areaType, gridWidth, gridHeight, position, connections, metadata, generatedBy
  - `buildings` table: id, areaId (FK), name, description, buildingType, floors, gridWidth, gridHeight, position, metadata, generatedBy
  - `rooms` table: characterId (NOT NULL), x, y, + nullable hierarchy columns (worldId, regionId, areaId, buildingId, floor). Unique on (characterId, x, y).
  - `stores` table: characterId (NOT NULL), roomX, roomY. Unique on (characterId, roomX, roomY). **No buildingId column yet.**
  - `npcs` table: characterId (NOT NULL), roomX, roomY. **No buildingId or areaId columns yet.**
  - `fogOfWar` table: characterId, entityType, entityId. Unique on (characterId, entityType, entityId).
  - `characters` table: worldId (nullable FK), currentFloor (nullable integer). **worldId is never set during character creation.**

  ### World Seed (`src/server/game/world-seed.ts`)
  - `seedStarterWorld()` creates: 1 world "Aethermoor", 3 regions (Ashen Coast, Verdant Vale, Iron Peaks), ~8 areas. **No buildings seeded yet.**
  - Idempotent — checks if Aethermoor exists before seeding.

  ### Character Creation (`src/server/trpc/routers/character.ts`)
  - `character.create` mutation inserts character but does NOT set worldId.
  - Calls `generateStartingRoom(characterId, theme)` which creates a room at (0,0) with no hierarchy fields.
  - New characters start with worldId === null (legacy mode).

  ### Store/NPC Lookup
  - Stores: looked up by `characterId + roomX + roomY`. Generated on first access via `generateStore()`.
  - NPCs: looked up by `characterId + roomX + roomY`. Generated on first access via `generateNPC()`.
  - Both are per-character — each character gets their own store/NPC instances.

  ### Navigation System (fully wired in S5+quickfix)
  - 7 tRPC endpoints: getWorldMap, getRegionMap, travelToRegion, travelToArea, enterBuilding, exitBuilding, changeFloor
  - `src/server/game/navigation.ts` — navigation logic with fog_of_war discovery
  - World gen pipeline: region/area/building generators in `src/server/game/world-gen/`
  - Frontend: WorldMapView, RegionMapView, TileMap, Breadcrumb, ContextDrawer all wired
  - Legacy check: `isWorldCharacter = !!player?.worldId` — controls MapPanel vs TileMap rendering

  ### Game Page
  - World characters: TileMap + pixel UI + Breadcrumb + ContextDrawer
  - Legacy characters: old MapPanel + terminal UI
  - Both paths coexist in `src/app/(game)/play/[characterId]/page.tsx`

  ## Key Design Decisions:

  - **Shared content**: Rooms, stores, NPCs in the hierarchy are shared across all players. Per-character data (fog, quests, combat, inventory, lore) stays isolated.
  - **Room lookup transition**: Legacy rooms use (characterId, x, y). Shared rooms use (areaId/buildingId, floor, x, y). Both patterns must work simultaneously.
  - **Migration is one-time**: Legacy characters get assigned to Aethermoor. Their existing rooms get hierarchy fields populated. After migration, they use the new system.
  - **Stores/NPCs shared economy**: If one player buys the last sword, it's gone for everyone. Restock mechanism prevents depletion.
  - **No real-time multiplayer**: Player presence is query-based (stale 30-60s), not websocket. Chat/trading/co-op is future work.
